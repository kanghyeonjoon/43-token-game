# 케어루프 CRM — 실무자 관점 개선 기획안 (v1.0)

> 프로토타입 v0.3을 **Tree-of-Thoughts 방식의 자기 검토**로 평가한 결과와 개선 로드맵.
> 4개의 독립된 가지(페르소나)가 각자 "하루 업무 시나리오"를 따라 화면을 검증하고, 발견을 심각도×빈도로 채점한 뒤 중복을 걷어내고 P0/P1/P2로 종합했다.
> **P0 13건은 v0.4에 즉시 반영 완료.**

---

## 1. 검토 방법 (ToT: 분기 → 평가 → 가지치기 → 종합)

| 가지 | 페르소나 | 검증 시나리오 |
|---|---|---|
| B1 | **데스크 실장** (52세, 병원 15년, IT 중급) | 출근→오늘의 액션→수납 완료 등록→전화 응대 조회→마감 EMR 대조 |
| B2 | **대표원장** (48세, 진료 사이 2~3분, 저녁엔 휴대폰) | 오늘 상태 판단→진료 전 환자 브리핑→위기 대응→월말 채널·매출 리뷰 |
| B3 | **신입 간호조무사** (26세, 입사 3일차, 인수인계 없음) | 첫 화면 이해→용어 장벽→실수 시나리오→환자 문의 응대→빈 화면 |
| B4 | **냉정한 제품 관점** (자기 검토) | 상태 영속성·데이터 정합성·모바일·빈 상태·프로토타입→실서비스 갭 |

평가 기준: **심각도**(차단 > 비효율 > 거슬림) × **빈도**(매일 > 주간 > 가끔). 매일×차단 = P0.

## 2. 각 가지가 내린 한 줄 결론

- **B1 (실장)**: *"환자를 완료 처리를 못 하겠다"* — 하루의 핵심 동작(수납 시 완료 등록) 버튼이 없고, 오전 내내 체크한 작업이 새로고침에 다 날아간다. "손이 한 일이 하나도 남지 않는 프로그램"이라는 인상 → 도입 저항.
- **B2 (원장)**: *결제를 유지시키는 결정적 화면은 '환자 360 — 진료 전 10초 브리핑'인데, 지금은 마케터를 위해 조립되어 있다* — 첫 카드가 관여도 점수이고 진료 기록이 맨 끝. '오늘'을 보여주는 원장용 요약도 없다.
- **B3 (신입)**: *"오늘의 액션 화면 자체가 인수인계서가 되어야 한다"* — 지시문("위기 상태 갱신")만 있고 실행 버튼이 없으며, 용어(관여도·D+5·발송 가드)에 설명이 하나도 없다. 결정적으로 **안전점검이 경고만 하고 실제로는 발송을 막지 않는다.**
- **B4 (제품)**: 경고와 동작의 불일치(가드), 화면 간 숫자 모순(비용 9,840원 vs 0원, 발송중단 4명 vs 목록 2명), 모바일 붕괴(사이드바 228px 고정) — 신뢰를 깎는 결함들.

## 4. 이 검토가 다시 세운 UX 원칙

1. **손이 한 일은 반드시 남는다** — 모든 체크·처리·상태 변경은 저장되고, 새로고침에 살아남는다.
2. **지시문이 아니라 버튼** — "~하세요"라고 말하는 모든 문장 옆에는 그 일을 끝내는 버튼이 있어야 한다.
3. **경고는 곧 차단** — 위험하다고 표시한 것은 시스템이 실제로 막는다. 경고와 동작이 다르면 없느니만 못하다.
4. **화면이 곧 인수인계서** — 3일차 신입이 위에서 아래로 읽으며 버튼만 눌러도 하루가 돌아가야 한다.
5. **숫자는 한 곳에서 계산한다** — 같은 값이 두 화면에서 다르면 제품 전체가 의심받는다.
6. **원장의 첫 질문은 "오늘 어때?"** — 30일 지표보다 오늘 밴드가 먼저다.
