# 튜브크루 — ToT 자기 검토 (v0.6)

> 튜브크루 v0.5를 **Tree-of-Thoughts 방식의 자기 검토**로 평가한 결과와 반영 내역.
> 4개의 독립된 가지(페르소나)가 각자 "실제 사용 시나리오"를 따라 화면과 산출물을 검증하고,
> 발견을 심각도×빈도로 채점한 뒤 중복을 걷어내 P0/P1/P2로 종합했다.
> **P0 2건 + P1 4건은 v0.6에 즉시 반영 완료.**

## 1. 검토 방법 (ToT: 분기 → 평가 → 가지치기 → 종합)

| 가지 | 페르소나 | 검증 시나리오 |
|---|---|---|
| B1 | **대행사 실장** (클라이언트 3채널 운영, 주 6편 납품) | 프로젝트 여러 개 관리 → 회의 → 납품물 클라이언트 전달 → 다음 편 |
| B2 | **초보 유튜버 대표** (1인 채널, 이런 도구 처음) | 데모 체험 → 키 연결 → 첫 실전 제작 → 회의록 읽기 |
| B3 | **콘텐츠 품질 관점** | 회의 발언·산출물이 실제 촬영/편집/업로드에 그대로 쓸 수 있는 수준인가 |
| B4 | **냉정한 제품/기술 관점** | 상태 영속성 · 데이터 정합 · 모드 전환 엣지 케이스 · 토큰 낭비 |

평가 기준: **심각도**(차단 > 비효율 > 거슬림) × **빈도**(매 편 > 주간 > 가끔).

## 2. 각 가지가 내린 한 줄 결론

- **B2 (초보 대표)**: *"회의록이 보고서 낭독회 같다"* — 전원이 ①②③ 번호를 매기며 완벽한 문단으로
  말하고, 발언마다 "○○ 기준으로"를 반복해 사람 회의처럼 안 읽힌다. (대표 지적과 일치 — P0)
- **B4 (제품)**: *"데모에서 라이브로 넘어가는 순간이 위험하다"* — 데모 산출물이 남은 채 키를 연결하고
  대본을 맡기면 `[데모 예시]` 기획안이 실제 제작 컨텍스트로 들어간다. 조용한 오염. (P0)
- **B1 (실장)**: *"납품을 못 꺼낸다"* — 단계별 복사만 있고 전체 내보내기가 없다. 클라이언트 전달은
  매 편 일어나는 일. 프로젝트 이름 변경도 없어 "새 프로젝트"가 쌓인다. (P1)
- **B3 (품질)**: 산출물 자체는 검수·자가 검증 게이트로 준수하나, [자가 검증] 블록이 다음 단계
  프롬프트에 그대로 딸려 들어가 토큰을 낭비하고 모델을 혼란시킬 여지. (P1)

## 4. 이 검토가 다시 세운 원칙

1. **회의록은 녹취록처럼 읽혀야 한다** — 정돈된 보고서는 신뢰를 깎는다. 사람은 번호 매기며 말하지 않는다.
2. **모드 전환의 이음새가 가장 위험하다** — 데모→라이브처럼 상태가 섞이는 지점은 시스템이 막는다.
3. **품질 게이트는 소비 지점에서 끝난다** — 검증 블록이 파이프라인을 타고 흐르면 노이즈가 된다.
4. **매 편 일어나는 일(납품·이름·이탈)이 P1이다** — 빈도가 심각도를 이긴다.
