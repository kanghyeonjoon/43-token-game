# 튜브크루 — ToT 자기 검토 (v0.6)

> 튜브크루 v0.5를 **Tree-of-Thoughts 방식의 자기 검토**로 평가한 결과와 반영 내역.
> 4개의 독립된 가지(페르소나)가 각자 "실제 사용 시나리오"를 따라 화면과 산출물을 검증하고,
> 발견을 심각도×빈도로 채점한 뒤 중복을 걷어내 P0/P1/P2로 종합했다.
> **P0 2건 + P1 4건은 v0.6에 즉시 반영 완료.**

## 1. 검토 방법 (ToT: 분기 → 평가 → 가지치기 → 종합)

| 가지 | 페르소나 | 검증 시나리오 |
|---|---|---|
| B1 | **대행사 실장** (클라이언트 3채널 운영, 주 6편 납품) | 프로젝트 여러 개 관리 → 회의 → 납품물 클라이언트 전달 → 다음 편 |
| B2 | **초보 유튜버 대표** (1인 채널, 이런 도구 처음) | 데모 체험 → 키 연결 → 첫 실전 제작 → 회의록 읽기 |
| B3 | **콘텐츠 품질 관점** | 회의 발언·산출물이 실제 촬영/편집/업로드에 그대로 쓸 수 있는 수준인가 |
| B4 | **냉정한 제품/기술 관점** | 상태 영속성 · 데이터 정합 · 모드 전환 엣지 케이스 · 토큰 낭비 |

평가 기준: **심각도**(차단 > 비효율 > 거슬림) × **빈도**(매 편 > 주간 > 가끔).

## 2. 각 가지가 내린 한 줄 결론

- **B2 (초보 대표)**: *"회의록이 보고서 낭독회 같다"* — 전원이 ①②③ 번호를 매기며 완벽한 문단으로
  말하고, 발언마다 "○○ 기준으로"를 반복해 사람 회의처럼 안 읽힌다. (대표 지적과 일치 — P0)
- **B4 (제품)**: *"데모에서 라이브로 넘어가는 순간이 위험하다"* — 데모 산출물이 남은 채 키를 연결하고
  대본을 맡기면 `[데모 예시]` 기획안이 실제 제작 컨텍스트로 들어간다. 조용한 오염. (P0)
- **B1 (실장)**: *"납품을 못 꺼낸다"* — 단계별 복사만 있고 전체 내보내기가 없다. 클라이언트 전달은
  매 편 일어나는 일. 프로젝트 이름 변경도 없어 "새 프로젝트"가 쌓인다. (P1)
- **B3 (품질)**: 산출물 자체는 검수·자가 검증 게이트로 준수하나, [자가 검증] 블록이 다음 단계
  프롬프트에 그대로 딸려 들어가 토큰을 낭비하고 모델을 혼란시킬 여지. (P1)

## 4. 이 검토가 다시 세운 원칙

1. **회의록은 녹취록처럼 읽혀야 한다** — 정돈된 보고서는 신뢰를 깎는다. 사람은 번호 매기며 말하지 않는다.
2. **모드 전환의 이음새가 가장 위험하다** — 데모→라이브처럼 상태가 섞이는 지점은 시스템이 막는다.
3. **품질 게이트는 소비 지점에서 끝난다** — 검증 블록이 파이프라인을 타고 흐르면 노이즈가 된다.
4. **매 편 일어나는 일(납품·이름·이탈)이 P1이다** — 빈도가 심각도를 이긴다.

## 5. v0.7 — 대표 직접 피드백 반영 (검토 라운드 밖 추가)

| 지적 | 반영 |
|---|---|
| "주제를 던져줬는데 주제부터 별론데, 이것부터 고를 수 없을까" | [💡 주제 발굴] — 기획 PD × 패키징 매니저 아이템 미팅이 유형이 전부 다른 후보 6개(절차/실수/비교검증/스토리/시의성/리스트)를 제출. 후보마다 📌왜 지금(PD)·📦클릭 각(패키징)·🔍주의점(검수) 표기. 고르면 주제 확정, 피드백 적고 [🔄 다시 발굴] 가능. 호출 1회 |
| "시작하면 정지를 못 하는데, 중간에 회의를 못 하게 할 수는 없나" | [⏹ 중지] — 회의·전체 실행·단계·검수·주제 발굴 전부 적용. `AbortController`로 진행 중인 API 호출을 즉시 끊고, 회의는 그때까지의 발언 유지·전체 실행은 완료 단계 유지. 파이프라인 단계 사이 요청도 놓치지 않게 중지 플래그를 파이프라인 단위로 소유 |

B2(초보 대표) 가지가 놓친 영역이었다 — 검토는 "만든 기능이 잘 동작하는가"를 봤지,
**시작 전(주제)과 진행 중(제어권)** 을 보지 않았다. 다음 라운드 체크리스트에
"개입 지점: 사용자가 멈추고 고칠 수 있는가"를 추가한다.

## 6. v1.4 — 안정화 라운드 (대표가 실행 불가한 상황에서 대리 검증)

**발견·수정 (P0급 1건):** 회의·제작이 진행되는 중에 프로젝트를 전환/삭제하면 `P()`가
다른 프로젝트를 가리켜 **진행 중 산출물이 엉뚱한 프로젝트에 기록되는 오염 버그** —
전환·새 프로젝트·삭제·이전 버전 복원·백업 복원을 작업 중에는 차단(먼저 ⏹ 중지 안내)으로 수정.
부수 보강: 스트리밍 중 산출물 칸 편집 잠금(readOnly), 손상 백업 파일 형태 검증.

**모의 API 라이브 총연습(키 없이 서버 응답을 재현해 라이브 코드 경로 전부 실행):**
회의 6콜(SCORES 파싱·본문 분리·스코어보드) → 전체 실행 5콜([자가 검증] 게이트 보존 +
다음 단계 프롬프트에 미전파 확인) → 업로드 판정 배지 → 검수 1콜 → 업무 1콜 = 13콜 집계 일치,
401 오류 시 우아한 실패(busy 해제·키 안내·산출물 무오염)까지 검증. **v0.6 원칙 2("모드 전환의
이음새가 가장 위험하다")의 재확인** — 이번에도 버그는 기능이 아니라 상태 전환(진행 중 ↔ 전환)에 있었다.
