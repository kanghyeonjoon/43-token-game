# 경희나비솔 한의원 CRM & 대시보드 기획서 (v3.0)

> 원안 메모(v2.0 구축 요청서 포함)를 5개 관점 — 규제·컴플라이언스 / 스코어링 로직 / 대시보드 UX·KPI / 기술 아키텍처 / 완결성 — 에서 전면 검토하여 52건의 수정·보완 사항을 반영한 개정 기획서입니다.
> 원안의 핵심 철학(환자 관여도 기반 1:1 밀착 케어, 스팸이 아닌 진심 어린 관리)은 그대로 유지하되, **그대로 구축하면 작동하지 않거나 위법한 부분**을 고치고, 원안에 없던 **대시보드 화면 정의**를 신설했습니다.

---

## 1. 프로젝트 개요

### 1.1 배경과 사명
- 25년 경력의 한의학 박사 원장이 1:1로 모든 환자를 직접 진료하는 클리닉.
- 대형 한의원의 마케팅 공세에 '온라인 광고'가 아닌 **'환자 중심의 압도적인 1:1 밀착 케어'**로 대응하여 경영 정상화를 목표로 함.
- 사명: "당신의 건강회복과 퍼포먼스 복원이 우리의 사명입니다."

### 1.2 핵심 목표
단순 매출 증대가 아닌 환자의 **자발적인 구전(WOM) 창출**:
- "여기 원장은 내 몸에 진심인 것 같아."
- "여기 다니고 (주증상 외에) 증상이 3개나 좋아졌어."
- "과잉진료 없고, 내가 선택할 수 있게 해줘."

### 1.3 사용자
| 사용자 | 목적 | 주 사용 화면 |
|---|---|---|
| **원장** | 경영 판단, 진료 전 환자 브리핑 | KPI 보드, 환자 360 뷰 |
| **데스크(실장)** | 오늘 해야 할 실행 업무 | 오늘의 액션 큐, 15일 처방 트래커 |

### 1.4 성공 기준 — 베이스라인 먼저 (원안에 없던 항목)
시스템 완성을 기다리지 말고 **착수와 동시에** EMR에서 직전 6~12개월 수치를 추출해 베이스라인으로 고정한다. 이것이 없으면 6개월 뒤 "효과가 있었는가"에 영원히 답할 수 없다.

| 지표 | 베이스라인(도입 전 측정) | 목표(6개월) |
|---|---|---|
| 초진 → 3회차 이상 재내원율 | EMR에서 추출 | +10%p |
| 치료 중단율(마지막 방문 후 30일 무예약) | EMR에서 추출 | -10%p |
| 한약 재처방률 | EMR에서 추출 | +15%p |
| 월 소개환자 수 | EMR/구두 확인 | +@ (측정 체계 신설) |
| 수신거부율 | — (신규 지표) | 1% 미만 유지 |

전면 가동 전 **4~8주 파일럿**(신규 초진 환자만 대상, 시나리오 3·4만 가동)을 두고, 파일럿 집단과 직전 동기 환자의 재내원율을 전/후 비교한다.

## 3. 핵심 전략: 환자 관여도 2-Factor 동적 분류 (개정)

### 3.1 기본 구조 (원안 유지)
```
[최종 관여도] = [Factor 1: 잠재성 점수] + [Factor 2: 활성도 점수(최근 90일)]
```
- 이론적 배경: PAM(환자 활성화 측정) + 리드 스코어링 — 원안과 동일.
- 인구통계가 아닌 **행동 데이터**로 점수화한다는 원칙 유지.

### 3.2 Factor 1: 잠재성 점수 (초진 문진표 기반)

| 항목 | 점수 | 비고 |
|---|---|---|
| 치료 목표 = [집중 진료] | +3 | 가장 강력한 신호 |
| 치료 목표 = [표준 진료] | 0 | |
| 치료 목표 = [최소 진료] | -3 | 명확한 관여도 낮음 신호 |
| 걱정증상 1개 이상 체크 | +2 | 예방 니즈 = PAM 높음 |
| 현재 다른증상 2개 이상 체크 | +1 | 종합 관리 니즈 |
| 가족증상 1개 이상 체크 | +1 | 잠재적 예방 니즈 |
| 진료실 적극 질문(근본 원인·예방·재발) | +1 | **DB1 체크박스로 채택 확정** — 원장이 진료 직후 태블릿에서 체크, 체크 시 Make가 잠재성 점수 재계산 |

> 원안의 "EMR에 체크"는 EMR 미연동 상태라 반영 경로가 없었음 → DB1 체크박스 + 재계산 시나리오로 실행 경로를 명시함. 재내원 시 간이 문진 재작성으로 잠재성 점수를 갱신할 수 있다.

### 3.3 Factor 2: 활성도 점수 (개정 — 최근 90일 롤링 합산)

원안의 치명적 문제 3가지를 수정했다:
- **'긍정 회신 +3'은 발생 불가** — 알림톡은 단방향 채널로 환자가 답장할 수 없다. → 버튼 클릭으로 대체.
- **'읽음 0점'은 측정 불가** — 읽음 데이터는 사업자에게 제공되지 않는다. → 삭제.
- **무한 누적** — 같은 링크 5번 클릭 = +5, 6개월 전 반응으로 영구 '상' 유지. → 발송 건당 1회 인정 + 90일 롤링.

| 이벤트 | 점수 | 비고 |
|---|---|---|
| '도움이 됐어요' 긍정 버튼 클릭 | +2 | 알림톡/친구톡 버튼 → Make 웹훅 |
| 콘텐츠 링크 클릭 | +1 | **발송 건(메시지ID)당 최대 1회** |
| D+5 경과체크 응답: 좋음 | +2 | |
| D+5 경과체크 응답: 보통 | +1 | |
| 재내원 완료 | +1/회 | **행동 점수 신설** — DB2 '완료' 등록 시 |
| 한약 처방 결제 | +3 | 행동 점수 신설 — DB3 등록 시 |
| 소개환자 발생 | +3 | 행동 점수 신설 — 소개자에게 가산 |
| 무반응 | 0 | |
| ~~부정 회신 -10 (Flag)~~ | **폐지** | 발송중단은 점수가 아닌 **독립 상태 필드**로 관리 (→ 3.6) |

> 행동 점수를 신설한 이유: 원안에서는 '최소 진료'를 골랐지만 이후 10회 재내원하고 한약까지 결제한 환자도 **영원히 '하'로 고착**되는 구조였다(하 그룹은 콘텐츠를 받지 않아 클릭할 기회 자체가 없음). 원안 128행의 "'플랜 결제' 시 승격"도 점수 항목에 없어 실행 경로가 없는 선언이었다.

### 3.4 그룹 분류 및 동적 규칙

| 그룹 | 기준(합산) | 케어 전략 |
|---|---|---|
| **상 (High)** | 4점 이상 | Full Sequence — 초·중·후기 콘텐츠 + 전 프로그램 |
| **중 (Medium)** | 1~3점 | Nurturing — 초기 + 중기(정보성 위주 1회), 반응 시 승격 |
| **하 (Low)** | 0점 이하 | Minimal — 기능성 메시지만 + 분기 1회 저부담 재평가 접점 |

- **승격**: 중기 콘텐츠에 긍정 반응(+2 이상 이벤트) 또는 한약 결제 시 즉시 '상' 승격, Full Sequence 가동 (원안 유지).
- **강등(신설)**: '상' 유지 조건 = 최근 60일 내 반응 또는 방문 1회 이상. 미충족 시 '중'으로 자동 강등. (6개월 무반응 환자에게 Full Sequence가 계속 나가 스팸 인상을 주는 것을 방지 — 원안이 막으려던 바로 그 상황.)
- **하 그룹 승격 경로(신설)**: 분기 1회 계절 건강 안내 1건 허용(월 상한 예외), 예약 알림 하단에 선택형 콘텐츠 링크 1개 허용, 재내원 시 간이 문진으로 잠재성 재평가.
- **수동 오버라이드(신설)**: DB1 `그룹 수동지정(자동/상/중/하)` 필드. 수동지정 ≠ '자동'이면 수동값이 우선. 지인·VIP, 컴플레인 이력, 보호자 대필 문진 등 원장 판단 반영용.
- 커트라인(4/1/0)은 파일럿 데이터로 보정한다(2단계 게이트 리뷰 항목).

### 3.5 최초 1회 발송 정책 (원안 모순 해소)
원안은 "모든 환자에게 최초 1회 CRM 발송"(93행)과 시나리오 1의 "'하' 그룹 제외"(250행)가 정면 모순이었다. 초진 시점에는 활성도=0이므로 하 그룹은 반응할 기회조차 없어 Factor 2가 무의미해진다.

**확정 정책**: ① 초기(주증상) 콘텐츠는 발송중단이 아닌 **전 환자에게 1회 발송**(활성도 측정의 씨앗), ② 중기 콘텐츠는 상+중 그룹, ③ 후기 콘텐츠는 상 그룹만.

### 3.6 발송중단(Flag) — 점수와 분리 (개정)
원안의 "-10점" 방식은 이후 긍정 반응이 누적되면 점수가 회복되어 **발송이 재개되는 사고**가 가능한 구조였고, DB에 플래그 필드 자체가 없었다.

- DB1에 `발송중단(Checkbox)` + `중단 사유` + `중단일` 독립 필드 신설. **모든 발송 시나리오의 1순위 공통 선행 조건**.
- 키워드 자동 감지("그만 보내세요" 등)는 **후보 탐지**로만 사용: 감지 시 해당 환자 발송을 '보류'로 안전측 전환 + Slack 알림 → 직원이 원문 확인 후 수동 확정. ("이제 안 보내주셔도 될 만큼 좋아졌어요, 감사합니다" 같은 문장의 오탐 방지.)
- 데스크에서 구두 요청받은 경우의 수동 중단 경로를 스태프 플레이북에 포함.
- 수신거부는 법적 차단 상태이므로 점수 재계산으로 절대 풀리지 않는다.

## 4. 채널 전략 (신설 — 원안의 기술 전제 교체)

### 4.1 3계층 채널 구조

| 계층 | 채널 | 대상 메시지 | 요건 |
|---|---|---|---|
| 1 | **알림톡** (정보성, 템플릿 사전 심사) | 예약 확인/변경, 복약지도, D+5 경과체크, 설문 참여 감사 등 거래·서비스 이행성 | 템플릿 사전 등록·심사(수일~2주) |
| 2 | **친구톡** (광고성 허용) | 건강팁, 이벤트, 재내원 유도, 안부 | 카카오채널 친구 추가 + 마케팅 수신동의 |
| 3 | **LMS/MMS 문자** (친구 미추가자 폴백) | 위와 동일한 광고성 | `(광고)` 표기 + 080 무료수신거부 번호 필수 |

- 알림톡 실패 시 문자 대체발송(솔라피 기능)을 기본 설정.
- 발송 시나리오는 채널을 직접 지정하지 않고 "메시지 성격(정보성/광고성) + 친구 여부 + 동의 여부"로 자동 분기.

### 4.2 클릭 추적 아키텍처 (자체 구축 — 솔라피 기본 기능 아님)
알림톡 버튼은 단순 웹링크(WL)이며 클릭 시 웹훅을 쏘는 기능이 없다. 모든 버튼/링크 URL을 **Make.com 커스텀 웹훅 리다이렉트**로 발급한다:

```
https://hook.make.com/xxxx?p={환자토큰}&c={콘텐츠태그}&a={액션}
  → Make: DB4에 클릭 기록 + 활성도 점수 갱신 (멱등키: 환자ID+콘텐츠태그+일자)
  → 302 리다이렉트 → 실제 콘텐츠 랜딩(영상 페이지)
```

- URL에 전화번호 원문 금지 — 발송 시 생성한 **난수 토큰**을 DB1에 저장해 매칭 (개인정보 노출·매칭 실패 원천 차단).
- D+5 '좋음/보통/불편함' 버튼도 동일 패턴, `a=` 값만 분기.
- '도움이 됐어요' 긍정 버튼을 콘텐츠 메시지 하단에 표준 탑재 (원안의 '긍정 회신'을 대체하는 활성도 신호).

### 4.3 메시지 분류표 (컴플라이언스 판정 기준)

## 5. 환자 여정 & 자동화 시나리오 12개 (7개 → 12개)

### 5.0 공통 발송 가드 (모든 시나리오 선행 조건 — 신설)
모든 발송 시나리오는 실행 직전 아래를 순서대로 검사한다. 하나라도 걸리면 발송하지 않거나 지연 큐로 보낸다.

1. `발송중단 = false` AND `발송부적합 = false` (사망/장기부재/가족공유폰)
2. `위기 상태 ∉ {위기, 대응중}` — 위기관리 중에는 기능성·위기 대응 메시지 외 전면 보류 (원안은 '불편함' 응답 이틀 뒤 재처방 안내가 나가는 구조였음)
3. 광고성 메시지면 `마케팅 수신동의 = Y`
4. 현재 시각 **08:00~20:50** — 야간이면 익일 오전으로 지연(웹훅 즉시발송 시나리오 포함. 원안 시나리오 7은 밤 11시 설문 제출 시 야간전송 위반이 자동 발생하는 구조였음)
5. 빈도 상한: 최근 7일 마케팅성 발송 0건 AND 월 상한(상 4건 / 중 2건 / 하 0건 — 기능성·위기 알림·분기 1회 재평가 접점은 제외). 초과 시 폐기하지 않고 '대기' 큐잉.
6. 중복 방지: DB4에 '발송예정' 레코드 선생성(멱등키: `환자ID + 발송단계 + 콘텐츠태그`) — 이미 존재하면 중단. 발송 성공 시 '성공' 갱신.

충돌 시 우선순위: **위기관리 > 기능성(예약·복약) > 처방 D+n 시퀀스 > 증상 콘텐츠 > 이벤트**.

**DB 5: 관여도 이력 (Score History)** — 신설
- 환자(Relation), 일시, 변경 사유(문진/클릭/버튼/방문/처방/감쇠/수동), 변화량, 변경 후 점수, 변경 후 그룹
- 승격·강등 흐름, 점수 추이 그래프, "중 그룹 중 4주 내 승격 비율" 같은 육성 KPI가 전부 이 로그에서 나온다.

### 6.3 트리거 설계 원칙 (원안 결함 수정)
- **Notion Formula/Rollup은 Make 트리거가 될 수 없다** (값 변화가 last_edited_time을 갱신하지 않음). 방문횟수·관여도그룹 등 계산값은 전부 **Make가 계산해 일반 속성에 기록**한다.
- 트리거를 'DB2 신규 완료 레코드' **단일 웹훅/워치로 통일**하고, 한 시나리오 안에서 방문횟수 +1 → 라우터 분기(1회차→시나리오 2, ≥3→시나리오 7, ≥4→시나리오 5). 레이스 컨디션 제거 + 시나리오 수 절감.
- 등호 트리거(`방문횟수=N`) 전면 폐지 → `≥N AND 해당 단계 로그 없음` 패턴 (노쇼·예약 포함 집계 오류, 하루 장애 시 영구 누락 방지).
- 전화번호 정규화 모듈을 모든 웹훅 시나리오 앞단에 공통 배치. 매칭 0건 또는 2건 이상이면 점수를 버리지 말고 **수동확인 큐 적재 + 내부 알림**.

## 7. 대시보드 화면 기획 (신설 — 이 기획서의 핵심 추가)

> 원안에는 대시보드 화면 정의가 한 줄도 없었다. 그대로 발주하면 '보이지 않는 자동화 파이프라인'만 납품된다.

### 7.1 화면 구성 (6개)

#### ① 오늘의 액션 (데스크 뷰 — MVP 최우선)
- **목적**: 데스크가 아침에 이 화면만 보고 하루 업무를 시작한다. Slack 알림은 보조 채널, **이 큐가 원본(source of truth)**.
- **구성**: 우선순위 정렬 작업 카드 리스트 — 각 카드에 [항목 / 환자명 / 기한 / 완료 체크].
  1. 🔴 위기 대응 (최상단 고정, 등록 후 경과 시간 표시, 24시간 초과 시 강조)
  2. D+5 경과체크 미응답 확인 / D+12(종료-3일) 재처방 안내 대상
  3. 3회차 진심동반자 안내 대상
  4. 치료중단 추정 팔로업 / 완료자 팔로업
  5. 수신거부·매칭실패 수동확인 큐
- 완료 체크 시 큐에서 제거되고 DB에 처리 기록이 남는다(미처리 항목은 다음날 아침 재노출).

#### ② 15일 처방 트래커 (데스크·원장 공용 — MVP)
- **칸반**: `복약지도 발송 → D+5 대기 → 응답 수신(좋음/보통/불편함) → 재처방 안내 대기 → 재처방 완료 / 미전환`
- 처방 유형별(15/30/45일) 체크포인트는 부록 C 매트릭스를 따름.
- 카드: 환자명, 처방 유형, 복용 경과일, D+5 응답 배지.

#### ③ 환자 360 상세 (공용 — MVP)
- **상단**: 이름 / 관여도 그룹 / 배지(위기·발송중단·발송부적합) / 치료 상태 / 방문 횟수·마지막 방문일
- **본문 1**: 잠재성·활성도 점수 현재값 + 변경 이력 그래프(DB5)
- **본문 2**: 발송 타임라인(DB4 역순 — 일시/콘텐츠/채널/반응)
- **본문 3**: **다음 예정 액션** — 언제 무엇이 자동 발송될 예정인지 (데스크 전화 응대 시 중복 안내 방지, 발송중단 요청 시 예약 발송 파악)
- **하단**: 진료기록(DB2)·처방기록(DB3)
- 데스크 전화 응대와 원장 진료 전 브리핑의 공통 진입점.

#### ④ 환자 파이프라인 (원장 뷰 — 2단계)
- 관여도 상/중/하/발송중단 그룹별 환자 수 분포(도넛/막대) + 주간 승격 n명·강등 n명 흐름(DB5 기반)
- '중 그룹'(핵심 육성 대상)의 4주 내 승격 비율 — 육성 전략 효과 검증 지표.

#### ⑤ 재내원·이탈 관리 (원장·데스크 — 2단계)
- 마지막 방문 경과일 구간(14/30/60/90일)별 환자 수 + 리스트
- 각 환자의 팔로업 발송·반응 여부, 치료 상태 표시.

#### ⑥ 발송 성과 (원장 뷰 — 3단계)
- 시나리오별·콘텐츠 태그별: 발송수 / 클릭률 / 긍정버튼율 / D+5 응답률 / **수신거부율 추이**(콘텐츠·빈도 과다의 조기경보)
- 이번 달 vs 지난 달 비교가 기본 뷰.

## 8. 규제·컴플라이언스 체크리스트 (신설)

> 원안은 기능은 정교하나 법적 전제조건이 통째로 빠져 있어, 그대로 구축 시 위법(무동의 민감정보 처리, 무동의 광고 발송)이거나 작동 불능(알림톡 반려)이었다.

### 8.1 문진표 동의 4블록 (부록 A에 전문)
1. **[필수] 개인정보 수집·이용 동의** — 진료·예약 목적, 보유기간 명시
2. **[필수] 민감정보(건강정보) 처리 동의** — 증상·가족력은 개인정보보호법 제23조 민감정보
3. **[선택] 마케팅 수집·이용 및 광고성 정보 수신동의** — "걱정증상·가족증상·치료목표를 맞춤 건강정보/이벤트 안내에 활용, 카카오톡·문자 수신, 링크 클릭 등 반응 정보 수집" 명시. **미동의 시 진료 불이익 없음 문구 필수.** 미동의자는 자동으로 기능성 메시지만 발송.
4. **[필수 고지] 처리위탁·국외이전** — 솔라피/카카오, Notion Labs(미국), Make/Celonis, HeyGen, 설문폼 업체 — 업체명·국가·항목·기간 명시

### 8.2 데이터 최소화·코드화
- 진료기록성 정보(주증상 상세, 처방 내용)는 **EMR에만** 보관. 외부 CRM DB에는 마케팅 세그먼트에 필요한 **코드화된 최소 데이터**만(증상명 → 콘텐츠 태그 코드).
- HeyGen에는 환자 실명 변수를 넘기지 않음(사전 제작 영상으로 원천 회피 — 9.2).
- 관여도 등급(상/중/하)은 내부 운영 용어로만 사용, 환자 노출 화면·문구에 절대 표시하지 않음.

### 8.3 광고성 메시지 3대 요건
- `(광고)` + 발신자명 표기, 080 무료수신거부 번호(문자), 채널 차단=수신거부 처리(친구톡)
- **야간(21:00~익일 08:00) 전송 금지** — 공통 발송 가드 4번으로 시스템 차단
- 수신거부 즉시 중단 + 14일 내 처리결과 통지 + **동의 2년 주기 재확인** 배치(연 1회 시나리오)

### 8.4 의료법 리스크 제거
- **소개 리워드 구조 제거** (의료법 27조 3항 환자 유인·알선 — 형사처벌 대상): 설문 참여 감사 선물(쌍화 티백)은 소개 여부와 무관한 소액 답례로 명확히 분리. 소개 카드에는 어떤 혜택도 연동하지 않음(소개자 기록은 유입 경로 파악용으로만).
- 급여 진료 본인부담금 할인·면제는 설계에서 원천 배제. 비급여 이벤트는 기간·대상·할인 전 가격 명시.
- 이벤트 콘텐츠는 발송 전 원장 법규 검토 승인 단계(Notion 상태값 '법규검토완료' 없으면 발송 시나리오 미작동).

## 9. 기술 아키텍처 & 로드맵

### 9.1 대시보드 아키텍처 — 단계적 접근 (확정: 웹 프로토타입 우선)
정적 웹페이지에서 Notion API를 직접 호출하는 것은 불가능하다(CORS 차단 + integration 토큰을 프런트에 심으면 전체 환자 민감정보 DB의 읽기·쓰기 권한이 노출됨). 또한 Notion API에는 집계(GROUP BY) 기능이 없다.

| 단계 | 대시보드 | 데이터 |
|---|---|---|
| **1단계 (다음 작업)** | **단일 HTML 프로토타입** — 기존 `index.html` 패턴, 7장의 6개 화면을 목(가상)데이터로 구현해 화면·지표 합의 | 목데이터 |
| 2단계 | 동일 HTML + 실데이터 | (a) Make가 매시간 집계 스냅샷 JSON을 정적 호스팅에 푸시, 또는 (b) **Supabase 미러**(Make가 Notion 기록 시 동시 upsert — 기존 토큰게임과 동일 스택이라 학습 비용 없음) 중 택일 |
| 3단계 | 성과 차트 고도화 | Supabase 집계 뷰 (Notion Charts로 부족 시) |

- 환자 실명 화면은 최소한의 접근 인증(비밀번호 또는 Cloudflare Access) 필수.
- Notion 내부 필터 뷰는 MVP 기간 데스크 작업 큐의 임시 대체재로 병행 가능.

### 9.2 AI 영상 — 사전 제작 라이브러리로 전환
원안(발송 건마다 HeyGen API로 환자별 영상 생성)은 ① 비용 월 15~50만원, ② 비동기 생성 수 분 지연(대기 로직 미설계), ③ 렌더 실패 시 발송 누락의 3중 문제.

- **증상별 사전 제작 영상 10~15개** (주증상 톱10 + 걱정증상 톱5)를 1회만 제작해 비공개 랜딩페이지에 호스팅.
- 개인화는 메시지 텍스트로: "[환자명]님, 문진표에 적으신 소화불량 기억났습니다" — 체감 개인화 효과의 대부분은 문구에서 나온다.
- 환자명이 음성에 들어가는 실시간 생성은 3단계에서 '상' 그룹 VIP 소수에게만 시범 적용, 그 경우에도 생성완료 콜백 → 발송 2단 구조 + 타임아웃 시 텍스트 대체 폴백.

### 9.3 3단계 구축 로드맵

**병렬 트랙 (즉시 착수)**: 카카오 비즈니스 채널 개설 → 발신프로필 등록 → MVP 필수 템플릿 5~8종(웰컴/복약지도/D+5 경과체크/재처방 안내/예약알림) 우선 심사 — 각 단계 수일~2주 소요, 이걸 모르면 "시스템은 완성됐는데 보낼 메시지가 없는" 공백기 발생. 템플릿 관리 대장(문구/변수/심사상태/승인일) 운영, 반려 대비 LMS 대체 경로 확보.

| 단계 | 기간 | 범위 | 게이트(검증 가설) |
|---|---|---|---|
| **1단계 MVP** | 4~6주 | 문진표 v2.0 + 잠재성 점수 + 시나리오 1·3·4(웰컴/복약 시퀀스/위기관리) + 대시보드 프로토타입(오늘의 액션·처방 트래커·환자 360, 목데이터) + 베이스라인 측정 + 기존 환자 백필 | D+5 응답률 30%+ / 데스크 일일 입력 유지 가능 / 재내원 개선 신호 |
| **2단계** | +4주 | 클릭 웹훅 활성도 점수 + 관여도 그룹 분기 + 시나리오 2·5·7·8·9·10·11·12 + 대시보드 실데이터 연동(파이프라인·재내원 화면) | 클릭 +1이 재내원과 상관 있는가(커트라인 보정) / 중→상 승격 실제 발생 여부 |
| **3단계** | 이후 | 발송 성과 차트 + 월별 이벤트(분기 3개 롤링 제작) + VIP 개인화 영상 시범 | 수신거부율 1% 미만 유지 / 소개환자 증가 |

각 단계 말 게이트 리뷰: 발송량·수신거부율·재내원율 확인 후 다음 단계 진행 결정. **월 고정비 상한 15만원** — 초과 요인(개인화 영상, Supabase 유료 플랜)은 게이트에서 승인.

### 9.4 안정성 설계 (원안 부재 — '조용한 발송 중단' 방지)
- 모든 외부 호출 모듈에 오류 핸들러: 429/5xx는 자동 재시도 3회(간격 10분+), 4xx는 스킵 + Slack 경보.
- Incomplete Executions 저장 활성화(실패 건 수동 재실행).
- **일일 헬스체크 시나리오**: 각 시나리오의 최근 실행 성공 여부 + 자동 비활성화 감지 → Slack 일일 요약. (Make는 오류 누적 시 시나리오가 자동 꺼질 수 있어, 방치하면 발송이 수 주간 조용히 멈춘다.)
- 솔라피 메시지ID·발송 상태를 DB4에 기록 → 실패 건 재발송 목록 추출 가능.
- 주 1회 DB 5개 CSV 자동 백업(별도 저장소).

### 9.5 월 예상 운영비 (가정: 월 신규 100명, 활성 발송 대상 400명, 월 발송 1,500건)

### 10.1 운영 R&R 표 (시스템 가동 후 매일 발생하는 수동 업무)

| 업무 | 담당 | 빈도/소요 | 기한(SLA) | 미처리 시 |
|---|---|---|---|---|
| 방문 기록 입력(체크아웃 폼, 20~30초/건) | 데스크 | 매 수납 시 | 당일 마감 전 | 다음날 아침 재알림 |
| EMR ↔ DB2 건수 정합성 대조 (5분) | 데스크 | 일 1회 마감 | 당일 | 헬스체크 경보 |
| DB3 처방 등록 + 복용 시작일 입력 | 한약 담당 | 처방 시 | 당일 | 시퀀스 미발송 |
| 위기 알림 대응 전화 | 데스크→원장 | 발생 시 | **30분 내** | 에스컬레이션(원장 직통) |
| 진료실 적극성 체크 | 원장 | 진료 직후 | 당일 | 점수 미반영(허용) |
| 3회차 동반자 안내 | 데스크 | 큐 발생 시 | 당일 | 다음 방문 시 재트리거 |
| 부정회신·매칭실패 수동확인 큐 | 실장 | 일 1회 | 24시간 | 발송 보류 유지(안전측) |
| 첫 개인화 발송 검수 | 실장 | 환자당 1회 | 발송 전 | 발송 대기 |

- 가동 첫 달은 시나리오 1·3·4만 켜서 일일 추가 업무량을 실측한 뒤 확장한다(직원 부담·저항 관리).
- 직원 공유 시 "이 시스템이 줄여주는 업무(수기 안내 전화, 누락 걱정)"를 명시해 도입 명분 확보.

### 10.2 콘텐츠 인벤토리 (신설 — 원안은 물량 미산정)
증상 택소노미를 닫힌 목록으로 확정(부록 B)해야 콘텐츠 물량이 유한해진다.

| 구분 | 오픈 전 필수 | 오픈 후 증분 |
|---|---|---|
| 주증상 영상/팁 | 톱5 | 톱6~10 (2단계) |
| 복약지도 | 3종(15/30/45일) | — |
| 알림톡 템플릿 | 5~8종 | 증상별 템플릿 주 단위 배치 등록 |
| 걱정증상 영상 | — | 톱5 (2단계) |
| 월별 이벤트 | — | 분기별 3개 롤링 제작 (3단계) |

- 콘텐츠 목록표(콘텐츠명/형식/담당/마감/상태)로 관리.
- **Fallback 규칙**: 환자의 증상 태그에 매칭되는 콘텐츠가 없으면 '공통 건강관리 콘텐츠'를 발송하고 내부 알림으로 제작 필요를 기록 (자동화의 조용한 실패 방지).

### 10.3 기존 환자 마이그레이션 (신설 — 원안은 '오늘 이후 초진'만 커버)
- EMR에서 기존 환자의 이름/연락처/마지막 방문일/누적 방문횟수를 추출해 DB1에 백필. `환자 유형=기존` + `이월 방문횟수` 기록.
- 방문 횟수 = 이월 횟수 + 신규 '완료' 레코드 수 → 10년 단골이 "처음 뵙는" 웰컴 문구를 받는 사고 차단(시나리오 1·7 트리거 조건에 환자 유형 반영).
- 기존 환자는 재내원 시 **간이 문진(걱정증상·치료목표 2문항)**으로 잠재성 점수를 소급 부여. 미작성 시 '중' 그룹 기본값 + 기능성 메시지만.

### 10.4 개인화 안전 규칙 (신설 — 오발송 1건이 '진심 케어'를 '소름 돋는 감시'로 뒤집는다)
- 환자명은 문진표 원문이 아닌 데스크 확정 표기 사용. 개인화 필드(이름·증상) 포함 발송은 환자당 첫 1회 데스크 검수 큐 경유.
- **민감 병명 금칙어 매핑**: 치매→'기억력·뇌 건강', 중풍→'혈관 건강', 암→'정기 검진' 등 완곡 표현 콘텐츠 제작 규칙.
- `발송부적합` 상태(사망/장기부재/가족공유폰)를 모든 시나리오 공통 선행 조건으로.
- 오발송 발생 시 사과·정정 스크립트를 스태프 플레이북 위기 시나리오에 추가.

### 10.5 발주 조건 (제작 업체 계약에 포함 — 신설)
1. **인수 테스트**: 시나리오별 테스트 케이스(정상 1건 + 엣지 1건 이상) 목록과 검수 완료 기준 명시. 예: "테스트 환자로 D+5 '불편함' 클릭 시 5분 내 내부 알림 수신".
2. **테스트 모드**: 테스트 환자 레코드 + 원장 개인 번호로만 발송되는 모드 필수 (개발 중 실환자 오발송 차단).
3. **계정 소유권**: Notion 워크스페이스·Make·솔라피 발신프로필·HeyGen 전부 **한의원 명의**로 개설, 업체는 게스트 권한 — 계약 종료 시 시스템·데이터 인질화 방지.
4. **유지보수**: 월 무상 수정 횟수, 장애 대응 시간, 산출물(시나리오 구조도, 관리자 매뉴얼) 명시.

### 10.6 데이터 운영 정책
- 직원별 개인 계정(공용 계정 금지), 데스크는 작업 큐 뷰만 접근하는 권한 분리.
- 퇴사 시 당일 계정 회수(플레이북 체크리스트).
- 발송중단·치료종결 후 일정 기간 경과 환자의 CRM 필드 파기 주기 확정, 연 1회 정리 루틴.

### 10.7 내부 준비 To-Do (원안 유지 + 보완)
원안의 Phase 1~3(점수 시스템 확정 / 여정 지도 / 스태프 플레이북 / 15일 처방 프로그램·가격 / 동반자 이벤트 / AI 아바타 촬영 / 콘텐츠 템플릿 / 가치공유 미팅 / 롤플레잉)은 유지하되:
- 동반자 이벤트에서 **소개 리워드 연동 제거** (8.4)
- AI 아바타 촬영은 **사전 제작 영상용 1회 촬영**으로 축소 (9.2)
- **추가**: 베이스라인 추출(1.4), 증상 택소노미 확정 워크숍(부록 B), 알림톡 채널·템플릿 심사 착수(9.3 병렬 트랙), 기존 환자 명단 추출(10.3)
